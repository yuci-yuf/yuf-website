# Feature Plan — Downloadable Invoice + Invoice in Confirmation Email

## 1. Goal

Give every paying participant a proper **invoice / payment receipt**:

1. **Download** the invoice (PDF) on the registration success page.
2. **Attach** the same PDF to the **existing confirmation email** — one email, invoice included
   (no separate invoice email).
3. Applies to **all new paid registrations**, plus an **admin action** to re-download or re-send the
   invoice for any existing confirmed registration.

Decisions (locked):
- Invoice number: **code-based** → `INV-<registrationCode>` (e.g. `INV-YUF26-XXXXXX`).
- Default document is a **simple Payment Receipt**; supplying a **GSTIN** (one config value) upgrades
  it to a full **Tax Invoice** with no code change.

---

## 2. Razorpay webhook setup (answer to "which Active Events?")

Our site uses Razorpay **Orders + Checkout** (`orders.create`), **not** Razorpay's hosted **Invoices**
product. Therefore:

| Event | Select? | Why |
|---|---|---|
| `payment.captured` | ✅ Required | Authoritative "payment succeeded → confirm registration". |
| `payment.failed` | ✅ Required | Releases the reserved seat when payment fails. |
| `order.paid` | ➕ Optional | Handled as a safety net alongside `payment.captured`. |
| `invoice.paid` | ❌ Not useful | Only fires for Razorpay's own hosted invoices, which we never create — it will never fire for our payments. Harmless to leave, but does nothing here. |

Also set: **Webhook URL** = `https://<domain>/api/payments/webhook`, **Secret** = value of
`RAZORPAY_WEBHOOK_SECRET`. Our PDF invoice is generated by our own app after `payment.captured`.

---

## 3. What already exists (no rebuild needed)

- **Resend email** is wired: `lib/email.ts` (`sendRegistrationEmail` → branded HTML/text + QR
  entry-pass PNG attachment) and `app/api/registrations/email/route.ts` (reads the confirmed reg via
  Admin SDK, called fire-and-forget from the form). The invoice just needs to be **attached here**.
- Pricing math: `computeInvoice(base)` → `{ base, platformFee, gst, total }` and `formatINR` in
  `lib/pricing.ts` (2% platform fee, 18% GST).
- The registration doc (written in `app/api/registrations/order/route.ts`) stores participant fields,
  `eventTitle`, `locationVenue`, `locationDate`, `registrationCode`, `amountPaid` (**total only**),
  `status`, and later `paymentId`/`paidAt`.

**Gap:** the itemized breakdown (base / platform / GST) is **not** stored, and there is **no PDF
library** in the project yet.

## 4. Implementation

### 4.1 Dependency
- Add **`@react-pdf/renderer`** (server-side PDF via `renderToBuffer`, no headless browser). Add to
  `serverExternalPackages` in `next.config.ts` if the build requires it. Fallback: `pdfkit` /
  `pdf-lib` if it misbehaves under Turbopack.

### 4.2 Persist the breakdown at order time
- `app/api/registrations/order/route.ts`: inside the transaction, compute `const inv =
  computeInvoice(base)` and also store `invoiceBase`, `platformFee`, `gst` next to
  `amountPaid: inv.total` (free path too). The invoice then always matches what was charged, even if
  the event fee is edited later.

### 4.3 Invoice data builder — `lib/invoice.ts` (new)
- `buildInvoiceData(reg)` → `{ invoiceNumber: 'INV-'+code, issueDate, seller, buyer, lineItem,
  base, platformFee, gst, total }`.
- Seller: name + address from `siteConfig.contact`; optional `INVOICE_GSTIN` + legal name →
  heading "Tax Invoice" when set, else "Payment Receipt".
- Fallback for legacy docs without the breakdown: scan `base` in `0..amountPaid` for the value where
  `computeInvoice(base).total === amountPaid` (bounded, cheap). Reuses `lib/pricing.ts`.

### 4.4 PDF renderer — `lib/invoice-pdf.ts` (new)
- `renderInvoicePdf(data): Promise<Buffer>` using `@react-pdf/renderer`: branded header (YUF colors +
  logo), seller/buyer blocks, line item (event + venue/date), itemized totals, **PAID** stamp with
  payment id + date, footer.

### 4.5 Attach to the confirmation email
- `lib/email.ts`: `sendRegistrationEmail` gains optional `invoicePdf?: Buffer` (+ `invoiceNumber`);
  push a second attachment `YUF-Invoice-<code>.pdf`; add "Your payment invoice is attached." to the
  HTML + text bodies.
- `app/api/registrations/email/route.ts`: after loading the confirmed reg → `buildInvoiceData` →
  `renderInvoicePdf` (in `try/catch` so the email still sends if PDF fails) → pass to
  `sendRegistrationEmail`. Skip PDF for free (`amountPaid <= 0`) registrations.

### 4.6 Download route — `app/api/registrations/[id]/invoice/route.ts` (new)
- `GET`, `runtime = "nodejs"`. Read reg via Admin SDK; require `status === "confirmed"`;
  **authorize** with `?code=` that must equal the stored `registrationCode` (blocks id enumeration /
  data leaks). Return the PDF with
  `Content-Disposition: attachment; filename="YUF-Invoice-<code>.pdf"`.

### 4.7 Success-page download button
- `components/public/RegistrationForm.tsx`: add `regId` state; change `finishSuccess(code,
  registrationId)` and update both call sites (free path + `confirmPayment`). In the success screen
  add a "Download invoice (PDF)" link → `/api/registrations/${regId}/invoice?code=${regCode}` (shown
  for paid events only).

### 4.8 Admin re-send / download
- `app/admin/(panel)/registrations/page.tsx` (+ helper in `lib/admin-data.ts`): on confirmed/paid
  rows add **Download invoice** (link with `?code=`) and **Re-send email** (POST
  `/api/registrations/email` `{ registrationId }` — now attaches the invoice).

## 6. Verification (end-to-end)

- Run a **single** `npm run dev`. Use Razorpay **test** keys.
- Register + pay for a paid event → success page shows **Download invoice (PDF)**; the PDF's
  base/platform/GST/total match the amount paid, numbered `INV-YUF26-XXXXXX`.
- Inbox: **one** confirmation email carrying **both** the QR entry pass and the invoice PDF.
- Free (₹0) event → no invoice/button (confirm desired behavior).
- Admin → Registrations: **Download invoice** and **Re-send email** work on a confirmed row.
- Download route rejects a wrong/missing `?code=`.

## 7. Open item

- **GSTIN / legal entity:** default is a plain Payment Receipt (no GSTIN). To issue a compliant GST
  **Tax Invoice**, provide the GSTIN + legal name (e.g. Youth United Council of India, or VVISION as
  the accounting partner named in the Terms) — it becomes a one-value config change.
